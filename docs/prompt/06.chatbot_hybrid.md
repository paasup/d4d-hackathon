# 챗봇 하이브리드 LLM (2번 항목 상세 설계)

배경: [docs/prompt/03.extend_options.md](03.extend_options.md) 2번 항목 결론 — "옵션 B(사전정의 매칭)를 핵심 로직(알람/명령 하달)에 유지하고, 장교용 챗봇 Q&A에만 옵션 A(RAG/LLM)를 하이브리드로 얹는다." 이 문서는 그 하이브리드 경계와 구현 범위를 확정한다.

## 하이브리드 경계 — 무엇을 절대 안 건드리는지

- `rule_engine.check_and_trigger` (탄약 20% 이하 → AlertEvent 트리거)는 LLM 미개입, 그대로 유지. 알림 발령은 결정론적이어야 하며, 라이브 데모 중 LLM 오탐/무응답으로 알림이 안 뜨는 리스크를 절대 만들지 않음.
- `/api/command`, `Command` 검증/하달 로직도 LLM 미개입.
- LLM은 오직 `POST /api/chat` (챗봇 Q&A) 한 곳에만 적용.

## 전략 (개정) — 전체 컨텍스트 주입 LLM 우선, 규칙 매칭은 장애 시 폴백

데이터 규모 재확인: 에지 상태는 20~30개 × 5개 필드(edge_id/ammo_pct/status/posture/alert_level) — 요약해도 수십 줄, 교본도 30줄 안팎. 합쳐도 LLM 컨텍스트 윈도우에 여유롭게 들어가는 "소규모 데이터"이므로, 굳이 질의 의도를 키워드로 좁히지 않고 **모든 질의를 LLM에 우선 전달**하며 교본 전문 + 현재 전체 에지 상태 스냅샷을 컨텍스트로 함께 주입해 자유 형식 질의응답을 지원하는 쪽으로 전환한다.

- 기존 키워드 매칭 4종(`_most_critical_edge`/`_resupply_status`/`_manual_guidance`/`_overall_summary`)을 "1차 응답 경로"에서 내리고, **LLM 실패(타임아웃/API 에러/키 누락/네트워크 차단) 시에만 개입하는 오프라인 폴백**으로 우선순위를 반전한다. 함수 자체는 재사용, 실행 순서만 바뀜.
- 장점: 정확한 키워드를 쓰지 않아도(예: "GOP-21 상태 어때?", "지금 경계 태세 올려야 하나?") 자연어 질의에 대응 가능 — 데모 시나리오에 없는 질문도 커버.
- 리스크: 모든 질의가 네트워크·LLM 응답 속도에 의존하게 됨 → 타임아웃(3~5초) + try/except로 감싸 실패 시 즉시 기존 규칙 기반 함수로 폴백하는 구조는 반드시 유지해야 라이브 데모 중 "먹통" 위험이 없음.

## 컨텍스트 구성 및 데이터 필터

- 교본: 항상 전문 그대로 주입(30줄, 필터 불필요).
- 에지 상태: 기본은 `store.get_all_states()` 전체를 `ammo_pct` 오름차순 정렬해 위험한 순서로 주입(LLM이 위험도 판단할 때 앞쪽에 위험 항목이 오도록). 20~30줄 수준이라 전체 주입도 부담 없음.
- 선택적 필터: 질의에 특정 edge_id 패턴(예: `[A-Z]+-\d+`)이 포함되면 해당 에지만 컨텍스트를 좁힘 — 목적은 토큰 절약이 아니라 LLM이 엉뚱한 에지를 참조해 답하는 환각 방지.
- 최근 알림(`store.get_alerts()`)도 포함 검토 — "최근에 무슨 경보 떴어?" 류 질의에 대응하려면 필요. 개수가 많아지면 최근 N건으로 제한.
- 진짜 임베딩 기반 유사도 검색·벡터DB(Qdrant 등)는 구현하지 않음 — 4번 항목(실증 인프라)과 동일하게 "로드맵 슬라이드로만 제시" 원칙 유지. 데이터가 작아 전체 주입이 검색보다 단순하고 안정적.

## 정합성 (피칭 멘트)

03번 문서에 이미 명시된 주의사항 재사용: 데모는 오픈 API(예: OpenAI, 또는 업스테이지 Solar)로 시뮬레이션하고, 피칭에서는 "실증 버전은 폐쇄망 내 사설 LLM(Ollama/Solar 등)으로 대체된다"고 명확히 구분해 설명 — "폐쇄망 데이터 주권" 핵심 테마와 모순되지 않게 함.

## 작업 범위

CLAUDE.md 절대 규칙 준수: `/api/chat` 엔드포인트 계약(`{query: str} → {answer: str}`) 변경 없음, 내부 구현만 변경. `shared/schemas.py` 변경 불필요.

1. **central-backend/requirements.txt**: LLM 클라이언트 라이브러리(예: `openai`) + `.env` 로드용 `python-dotenv` 추가.
2. **central-backend/.env** (신규, git 미포함): `OPENAI_API_KEY` 등. **`.gitignore`에 `.env` 항목이 없으므로 먼저 추가** — 현재는 `.venv/`, `venv/`만 등록돼 있어 키가 실수로 커밋될 위험이 있음.
3. **central-backend/chat_engine.py**:
   - `answer(query)`를 `_llm_answer(query)` 우선 호출로 변경. 성공하면 그 결과를 그대로 반환.
   - `_llm_answer` 실패(예외/타임아웃) 시에만 기존 4개 키워드 매칭 함수로 폴백, 그마저 매칭 안 되면 최종 고정 문구("질문을 이해하지 못했습니다...") 반환. 기존 함수(`_most_critical_edge` 등)는 그대로 재사용.
   - 신규 `_build_context(query)`: 교본 전문 + `store.get_all_states()`를 `ammo_pct` 오름차순 정렬한 요약 + (질의에 edge_id 패턴이 있으면 해당 에지만 필터) + 최근 알림 N건을 조립.
   - 신규 `_llm_answer(query)`: `_build_context(query)` 결과를 시스템 프롬프트로 LLM 호출, 타임아웃 3~5초 + try/except.
4. **scripts/demo_scenario.py**: 리허설에 "기존 키워드 목록에 없는 자유 질의" 1건(예: "동쪽 에지들 상태 요약해줘")을 추가해 LLM 경로를 데모에서 실제로 보여줌.

## 완료 기준

- 자유 질의(예: 특정 edge_id 지정, 복합 조건, 데모 시나리오에 없던 표현)에 LLM이 교본 + 현재 상태 기반 자연어 답변을 생성.
- 기존 4개 키워드 질의(가장 위험한 에지는? / 보급 현황은? / 대응책은? / 전체 현황은?)도 LLM 경로를 통해 정상 응답(내용이 기존과 달라도 무방, 정확성만 유지).
- API 키를 지우거나 네트워크를 차단한 상태에서는 자동으로 기존 키워드 매칭 함수로 폴백해 챗봇이 에러 없이 응답 (데모 중 인터넷이 끊겨도 챗봇 자체는 안 죽음).
