# 원격 명령 확장 (RESUPPLY 외 5종)

배경: 03.extend_options.md(삭제됨) 3번 항목 + 데모 시나리오 4단계("다중 명령 하달")의 상세 설계. 별도 문서로 분리한 이유는 구현 범위가 schemas/edge-simulator/central-backend/dashboard/demo_scenario 5개 컴포넌트에 걸쳐 있어 단일 구현 세션의 입력 문서로 쓰기 위함.

## 복잡도 검토 결과 — 단순화 적용됨

이전 버전(v2)은 "정찰 결과"와 "보급 COP 이동"을 각각 새 WS `kind`(`recon_result`, `convoy`)로 만들고 중앙에 새 비동기 브로드캐스트 루프를 2개 추가하는 설계였음. 실제로 짜보면:
- central-backend에 새 백그라운드 태스크 2종 + 새 kind 2종 추가
- dashboard에 새 kind 2종을 처리하는 렌더링 로직 2종 추가

→ 남은 시간 대비 백엔드 변경 폭이 과했음. **동일한 시연 효과를 내면서 아래처럼 잘라냄**:

| 잘라낸 것 | 대체 방법 | 결과 |
|---|---|---|
| 새 kind `recon_result` + 중앙 타이머 브로드캐스트 | 기존 `AlertEvent`/`alert` kind를 그대로 재사용 (정찰 결과도 "알림"의 일종으로 취급) | central-backend 변경량 0에 가까움, dashboard 코드 변경 없음(이미 알림 렌더링 존재) |
| 새 kind `convoy` + 중앙의 좌표 보간 브로드캐스트 루프 | 보급 COP 이동 애니메이션을 **프론트엔드 전용**으로 처리 — "보급 실행" 버튼 클릭 시점에 depot 고정 좌표(프론트 상수)→선택된 에지 좌표를 10초짜리 CSS/JS 애니메이션으로 이동 | central-backend 변경 0, WS 프로토콜 변경 0 |

결과적으로 **central-backend는 RECON_DRONE 분기 하나만 추가**하면 되고(그 외 4개 명령은 기존 범용 relay 그대로 사용), 새로운 WS 메시지 형식도 생기지 않는다. 이 정도가 남은 시간 대비 적정 범위로 판단됨.

## 최종 명령 6종

| 명령 | 전달 대상 | 동작 |
|---|---|---|
| `RECON_DRONE` | 에지로 전달 안 함, 중앙에서만 처리 | 하달 후 ~6초 뒤 중앙이 `AlertEvent`(level="warning" 등, message="정찰 결과: ...")를 생성해 기존 `alert` kind로 브로드캐스트 + `store.add_alert` |
| `RESUPPLY` | 에지 (기존과 동일) | 에지: 변경 없음(10초 후 ammo_pct=100, status="resupplied"). 대시보드: 버튼 클릭 시 프론트엔드에서만 depot→에지 이동 애니메이션 재생(백엔드 관여 없음) |
| `SILENT_MODE` ↔ `ACTIVE_MODE` | 에지 (기존 `/ws/command` 그대로) | `posture`를 "silent"/"active"로 전환. silent면 상태 송신 주기를 5~10초 → 60초 이상으로 늘림 |
| `ALERT_LEVEL_UP` ↔ `ALERT_LEVEL_DOWN` | 에지 (기존 `/ws/command` 그대로) | `alert_level`을 "heightened"/"normal"로 전환. heightened면 탄약 소모 속도 2배 |

## 작업 범위

CLAUDE.md 절대 규칙 준수: `shared/schemas.py` 모델을 그대로 import, in-memory 유지. 엔드포인트 계약(`/api/command`, `/ws/command`, `/ws/dashboard`)은 **변경 없음** — 기존 kind(`state`/`alert`)만 재사용하므로 CLAUDE.md 엔드포인트 계약 표 수정도 불필요.

1. **shared/schemas.py**
   - `Command.command` Literal에 `"RECON_DRONE"`, `"SILENT_MODE"`, `"ACTIVE_MODE"`, `"ALERT_LEVEL_UP"`, `"ALERT_LEVEL_DOWN"` 추가 (`"RESUPPLY"` 유지)
   - `EdgeState`에 `posture: Literal["active", "silent"] = "active"`, `alert_level: Literal["normal", "heightened"] = "normal"` 필드 추가. 기존 `status`(idle/resupplying/resupplied/alert)는 변경 없음

2. **edge-simulator/agent.py**
   - `self.posture`, `self.alert_level` 초기값 추가, `_send_state`에 포함
   - `_listen_commands()`에 SILENT_MODE/ACTIVE_MODE/ALERT_LEVEL_UP/ALERT_LEVEL_DOWN 4개 분기 추가 (RECON_DRONE 분기는 만들지 않음 — 애초에 에지로 전달되지 않음)
   - `_run_state_loop`의 `asyncio.sleep(random.uniform(*config.SEND_INTERVAL_RANGE))` 부분을 posture가 "silent"면 훨씬 긴 간격(예: 60~90초)을 쓰도록 분기
   - `_simulate_ammo_drain`의 `config.AMMO_DRAIN_RANGE` 적용 부분을 alert_level이 "heightened"면 2배로 적용하도록 분기

3. **central-backend/main.py**
   - `/api/command`에서 `cmd.command == "RECON_DRONE"`일 때만 `manager.send_command`를 호출하지 **않고**, 대신 `asyncio.create_task`로 ~6초 후 `AlertEvent`를 만들어 `store.add_alert` + `manager.broadcast("alert", ...)` 실행 (rule_engine과 동일한 패턴 재사용, `level`/`message`는 랜덤 또는 고정 문구로 "정찰 결과: 보급로 안전 확인" / "정찰 결과: 적 활동 포착, 우회로 권장" 중 선택)
   - 그 외 5개 명령(RESUPPLY 포함)은 기존 범용 relay 그대로 사용 — **코드 변경 없음**
   - rule_engine.py, store.py 변경 불필요

4. **dashboard-web**
   - 기존 "RESUPPLY 하달" 버튼(app.js:96-121)을 참고해 버튼 4개 구성: 정찰 드론 투입(RECON_DRONE), 보급 실행(RESUPPLY, 클릭 시 프론트 전용 이동 애니메이션 트리거), 무선 침묵 전환/해제(현재 posture 값 보고 SILENT_MODE 또는 ACTIVE_MODE 하달), 경계태세 격상/격하(현재 alert_level 값 보고 ALERT_LEVEL_UP 또는 ALERT_LEVEL_DOWN 하달)
   - depot 고정 좌표를 프론트 상수로 하나 정의(예: 대대 보급기지 위치), RESUPPLY 클릭 시 Leaflet 마커를 depot 좌표에서 선택된 에지 좌표까지 10초간 `requestAnimationFrame` 또는 `setInterval`로 보간 이동시키는 함수 추가 (신규 WS 메시지 불필요, 순수 클라이언트 로직)
   - 지도 마커에 posture(silent=반투명/회색), alert_level(heightened=주황 테두리)을 기존 status 색상과 별개 오버레이로 표시
   - RECON_DRONE 결과는 기존 알림 패널(`renderAlert`, app.js:124~)이 그대로 처리하므로 추가 코드 불필요

5. **scripts/demo_scenario.py**
   - 리허설 시나리오를 "탄약 경고 → RECON_DRONE 하달 → 정찰 결과 알림 확인 → RESUPPLY 하달(보급 COP 이동 애니메이션 확인) → SILENT_MODE → ACTIVE_MODE → ALERT_LEVEL_UP → ALERT_LEVEL_DOWN" 순서로 갱신

## 완료 기준

대시보드에서 에지 하나를 선택해 "정찰 드론 투입" → 몇 초 후 알림 패널에 정찰 결과가 뜨는 것 확인 → "보급 실행" → 지도에서 보급 COP 마커가 depot에서 해당 에지까지 이동하는 애니메이션과 10초 후 탄약 100% 복귀 확인. 이어서 무선 침묵/해제, 경계 격상/격하를 각각 토글해 posture·alert_level이 실시간으로 바뀌고 송신 주기·탄약 소모 속도에 실제 반영되는 것을 눈으로 확인할 수 있어야 한다.

## 구현 세션 후 추가 반영 사항

위 설계대로 1차 구현한 뒤, 실제 기동/리허설 중 발견된 문제와 추가 요청을 이어서 반영함.

### 버그 수정

| 항목 | 문제 | 조치 |
|---|---|---|
| RECON_DRONE 알림 식별 불가 | 알림 메시지에 `edge_id`가 없어 에지가 여러 개일 때 어느 에지의 정찰 결과인지 구분 불가 | 메시지에 `[edge_id]` 접두 추가 (`central-backend/main.py` `_run_recon_drone`) |
| 경계태세 격상이 무효화됨 | `_simulate_ammo_drain`이 `status=="idle"`일 때만 동작해서, 이미 `alert` 상태(탄약≤20%)인 에지는 `ALERT_LEVEL_UP`을 걸어도 소모 속도가 그대로였음 | 드레인 조건을 `status in ("idle", "alert")`로 확장 (`edge-simulator/agent.py`) |
| 보급 후 탄약 영구 고정 | `resupplied` 상태로 전이된 뒤 다시 `idle`로 복귀하는 코드가 없어, 한 번 보급받은 에지는 이후 영원히 탄약 100%로 고정되고 재드레인/재알림이 발생하지 않음 | `resupplied`를 3초 노출한 뒤 `idle`로 복귀시켜 드레인 재개 (`edge-simulator/agent.py` `_handle_resupply`) |
| 지도 마커 소실/오표시 | Leaflet.markercluster는 `setLatLng`만으로 내부 공간 인덱스를 갱신하지 않아, 위치가 크게 바뀌면 마커가 잘못된 클러스터에 남아 화면에 안 보이는 현상 발생 | `upsertMarker`에서 `removeLayer` → `setLatLng` → `addLayer` 순으로 재삽입하도록 변경 (`dashboard-web/app.js`) |
| `/ws/command` 경합 조건 | 연결이 끊길 때 자기 자신이 등록한 커넥션이 아니어도 `edge_connections`에서 무조건 `pop`해서, 오래된(중복) 커넥션이 나중에 끊기며 방금 등록된 새 커넥션까지 지워버림 → 명령 하달이 조용히 실패 | 현재 등록된 커넥션이 자기 자신일 때만 `pop`하도록 identity 체크 추가 (`central-backend/main.py` `ws_command`) |
| 에지 위치 비결정성 | edge-simulator 재기동마다 좌표가 랜덤 재생성되어, 재기동할 때마다 대시보드 마커가 "이동"한 것처럼 보임 | 위치 생성에만 고정 시드(`random.Random(42)`) 적용 — 탄약 초기값(`AMMO_INIT_RANGE`)은 기존처럼 시드 없는 랜덤 유지 (`edge-simulator/main.py`) |

### UX 개선

- 정찰 결과를 알림 패널뿐 아니라 선택된 에지의 명령 버튼 영역에도 표시(`#d-recon-status`). 위험 판정("우회로 권장")이면 보급 실행 버튼에 경고 테두리를 씌우고 클릭 시 `confirm()`으로 한 번 더 확인
- "무선 상태" 상세 정보에 다음 상태 갱신까지 남은 시간을 추정 카운트다운으로 표시(매초 감소). 실제 송신 간격은 랜덤이라 근사치(active ~8초, silent ~75초)이며, 해당 에지의 실제 `timestamp`가 갱신되면 리셋됨
- 선택된 에지를 지도 위 별도 레이어에 검은색 펄스 링으로 하이라이트해, 클러스터링에 가려지지 않고 항상 식별 가능하게 표시
