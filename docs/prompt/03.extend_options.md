## 데모 목적 및 핵심 테마

**전반적 목적**: 이 데모는 단순한 해커톤용 아이디어가 아니라, 파스업(PAASUP)이 이미 추진 중인 R&D 로드맵의 축소 재현(Proof of Concept)이다.
근거 (파스업 브레인 지식 참고: `paasup-brain/knowledge/기업프로필.md`, `paasup-brain/knowledge/scoring_profile.md`):

- 파스업의 정체성: "내부망(폐쇄망) 환경에서 SaaS 수준의 데이터 인텔리전스 플랫폼을 제공하여 데이터 주권·AI 소버린티를 실현"하는 회사(PAASUP DIP).
- 2026-04 디딤돌 R&D 과제 협약: "세계 최고 수준의 폐쇄망 환경 전용 고밀도 **에지 데이터 인텔리전스 플랫폼(Edge DIP)**" 개발·상용화.
- 2026-06 모두의 챌린지 AX 과제: 업스테이지와 **국방 특화 LLM Agent** 공동 개발 협약.
- 즉, 이 프로젝트의 "에지 COP ↔ 중앙 플랫폼" 구조는 파스업이 실제로 진행 중인 Edge DIP + 국방 특화 LLM Agent 로드맵을 24시간 안에 눈으로 보여주는 시연판이다.

**핵심 테마 제안**: *"폐쇄망(에어갭)에서도 끊기지 않는 지휘통제 순환 — Data & AI Sovereignty for the Battlefield"*

- 메시지의 중심은 지도/알림 UI 자체가 아니라 "외부 클라우드·인터넷 없이도 에지-중앙 간 실시간 데이터·명령·AI 추천이 순환하는 자기완결형 국방 플랫폼"이라는 점.
- D4D 심사 기준 중 가장 배점이 큰 **군 적용 가능성(30%)**과 정면으로 부합함 — 실제 야전/폐쇄망 환경은 인터넷이 끊긴 경우가 대부분이므로 "폐쇄망에서도 동작"은 강력한 차별화 포인트.
- 파스업은 이미 정부 R&D 과제(디딤돌), Tesoro VC 투자, NVIDIA Inception 파트너십을 보유한 실체가 있는 회사이므로, "해커톤에서만 존재하는 아이디어"가 아니라 "이미 투자·과제로 검증되고 있는 로드맵의 프로토타입"이라는 신뢰도를 심사위원에게 전달할 수 있음 (Main Stage 진출 시 Founder & Team Drive, Global Readiness 배점에도 유리).

**정합성 주의**: 2번 항목(챗봇 하이브리드 LLM)에서 데모 편의상 외부 API를 쓰더라도, 피칭 멘트에서는 "실제 상용 버전은 폐쇄망 내 사설 LLM(Ollama/업스테이지 Solar 등)으로 대체되며, 이번 데모는 개발 시간 제약상 오픈 API로 시뮬레이션한 것"이라고 명확히 구분해 설명해야 "폐쇄망 데이터 주권" 테마와 모순되지 않는다. (4번 항목의 "Open API 이용한 시뮬레이션" 표현과 같은 논리)

## 데모 시나리오 (3분 발표 기준 흐름)

1. **오프닝 (~30초)**: 지도 위에 다수 에지(GOP 분초, K9 자주포 진지 등)가 5~10초 주기로 실시간 상태를 쏘는 화면. "이 구조 전체가 인터넷 연결 없이 로컬 네트워크만으로 동작합니다" 멘트로 폐쇄망 테마를 각인.
2. **위기 발생 (~30초)**: 특정 에지 탄약이 20% 이하로 급감(`/api/debug/set_ammo` 백도어로 즉시 재현) → 대시보드 경고 알림 + 룰엔진이 매칭한 교범 조항 권고문 팝업.
3. **지휘관 의사결정 지원 (~30초)**: 챗봇 창에 "지금 가장 위험한 에지는 어디고 대응책은?" 질의 → AI가 즉시 상황 요약 + 교범 기반 답변 제공.
4. **다중 명령 하달 (~60초, 3번 항목 신규 기능 핵심 시연 구간)**: 상황별로 명령을 선택 하달 — RESUPPLY(보급) / SILENT_MODE(무선침묵, 피탐 회피) / RECON_DRONE(정찰 드론 투입) / ALERT_LEVEL_UP(경계태세 격상). 명령을 누를 때마다 지도 마커 색과 사이드바 상태가 실시간으로 바뀌는 것을 보여줘 "상황별 다양한 지휘통제 옵션"이라는 창의성·군 적용 가능성 포인트를 동시에 어필.
5. **마무리 & 로드맵 (~30초)**: "지금 보신 것은 파스업이 디딤돌 R&D 과제로 개발 중인 Edge DIP와 업스테이지 협업 국방 특화 LLM Agent의 축소 프로토타입입니다. 실제 상용 버전은 Kafka-Starrocks-S3 기반 대용량 처리와 Qdrant+사설 LLM 기반 RAG로 확장됩니다" — 4번 항목을 로드맵 슬라이드로 제시.

---

1. 에지 데이터 스토리지의 유연성 (Schema)

[옵션 A] 메트릭 맵(metrics: dict[str, float]) 형태의 동적 스키마를 도입하여 연료, 배터리, 통신 상태 등 다양한 수치를 제약 없이 수집할 수 있게 설계
[옵션 B] 군수(탄약, 연료), 감시(적 식별 수), 작전(경계 등급) 등 몇 가지 핵심 지표 필드를 정적으로 설계

2. 지능형 추천 엔진 구현 방식

[옵션 A] 경량 로컬 LLM(Ollama 등) 또는 프라이빗 LLM API를 연계하여 상황에 맞는 교본/매뉴얼 구절을 검색하고 상황 맞춤형 지휘관 대응 권고문(RAG)을 실시간 생성
[옵션 B] 다차원 지표(예: 탄약 < 20% AND 적 출현 > 3) 조건에 반응하여 사전 정의된 대응 매뉴얼 구절을 매칭하는 구조로 설계 (성능 및 동작 신뢰성 우선)

3. 추가할 원격 명령 종류

현재 RESUPPLY(보급) 외에 시연에서 추가로 다루고자 하는 명령(예: SILENT_MODE 절전/은무 전환, RECON_DRONE 드론 정찰 지시, ALERT_LEVEL_UP 경계태세 격상 등)의 범위 선정

4. 실증 인프라 환경 적용

4.1 중앙COP: Kafka - Starrocks - S3 적용 / RAG (Qdrant) + Private LLM 적용 (Open API 이용한 시뮬레이션)   
4.2 에지COP: mqtt - Starrocks - s3 적용 / 에지 <-> Kafaka 통한 중앙COP 연계

5. 에지 영상 전송
5.1 에지 영상 실시간 에지COP 및 중앙COP 전달 체계. 네트워크 상황에 따라 중앙COP에는 영상 해상도 > 메시지 전달 체계까지 가능

6. OSINT 정보 분석
중앙COP 는 에지COP 지역의 OSINT 정보를 분석하고 주요 관련 정보를 중앙COP 의 추천 메시지로 제시 > 중앙 COP 에서 승인한 정보는 에지 COP 에 전달하는 체계

7. 시계열 정보 분석 
중앙COP 에 전달된 에지COP 의 위치 이동 경로를 시뮬레이션, 미래 이동 경로까지 시뮬레이션 하는 기능

8. 목표 중심 액션 플랜 제안
개별 아이템 상태 정보와 교본 중심의 기본 응대를 넘어, 작전 목표를 실행하기 위한 방법론을 제안하고, 실행 추적하는 기능

### 검토 결과 (1~4번 항목)

D4D 해커톤(https://luma.com/2ew4xn7b) 트랙 T3(Battle Network, C2 & Sustainment)에 해당. 1차 심사(현장 테이블 순회 피칭, 3분 발표+2분 QA) 배점은 군 적용 가능성 30% / 문제 적합성 25% / 기술 구현 25% / 창의성 20%, 제출 마감은 7/5(일) 10:00. 이 기준을 반영한 항목별 결론은 다음과 같음.

- **1번 (스토리지 유연성)** → [옵션 B] 유지 권장. 이미 정적 스키마로 구현돼 있고, 3번 항목에서 새 지표가 필요해지면 그때 필드를 개별 추가. 동적 dict(`metrics: dict[str, float]`) 전환은 남은 시간 대비 실익이 적음.
- **2번 (추천 엔진)** → [옵션 B]를 핵심 로직(알람/명령 하달)에 유지하고, 장교용 챗봇 Q&A에만 [옵션 A](RAG/LLM)를 하이브리드로 얹는 방식 권장. 신뢰성이 중요한 알람 트리거를 LLM으로 교체하면 라이브 데모 중 실패 리스크가 크고, 챗봇 한정 적용만으로도 창의성 점수는 확보 가능.
- **3번 (명령 종류 추가)** → 최우선 진행 권장. SILENT_MODE, RECON_DRONE, ALERT_LEVEL_UP 등 2~3개 추가. 저비용으로 군 적용 가능성·창의성 점수를 동시에 높일 수 있음.
- **4번 (실증 인프라)** → 실제 구현 대신 피칭 슬라이드의 "실증 로드맵"으로 제시 권장. 군 적용 가능성(30%)과 2단계(Main Stage) 진출 시 Global Readiness(25%) 점수를 낮은 리스크로 확보 가능. 남은 시간에 Kafka/Starrocks/S3/Qdrant/mqtt를 실제로 붙이면 CLAUDE.md의 "DB 세팅에 시간 쓰지 않는다" 원칙과 충돌하고 데모 신뢰성만 떨어뜨릴 위험이 큼.

권장 작업 순서: **3번 → UI 개선 → 2번(챗봇 하이브리드) → 4번(슬라이드화)**. 1번은 3번 진행 중 필요한 필드만 최소로 추가.
